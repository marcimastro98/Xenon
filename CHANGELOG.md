## [Unreleased]
### ✨ Xenon AI — richer dashboards it can actually build
- **Ask Xenon AI to "use tabs" and it now does**: when you ask it to build a page and group things into tabs — or even when you don't, but two widgets clearly belong together — it can merge them into a single tabbed tile (e.g. OBS and Twitch as tabs in one tile on a streaming page, or Tasks/Agenda/Notes together on a work page). Until now the assistant could only lay widgets out side by side, so a request like *"create a streaming page and use tabs where it makes sense"* was quietly ignored; it now composes the page the way you asked.
- **It can give a page a clear focal point**: the assistant can make the most important tile wider than the rest (for example the OBS preview on a streaming page, or the player on a music page) instead of always splitting the row evenly, so AI-built pages look intentional rather than uniform.
- **It can put the same widget on more than one page**: ask to keep the media player, mic or your tasks available across pages and the assistant mirrors that widget onto the other page as a live copy — the same "duplicate" you can do by hand, now voice/chat-driven.
- **When it sets up a Deck for you, it can wire any kind of key**: the assistant can now build Deck keys for the full range of actions the Deck itself supports — open an app/file/website, run a hotkey or a webhook, play a soundboard clip, adjust a single app's volume, switch OBS scenes or toggle recording/streaming, Twitch/YouTube/Streamer.bot actions, and more — instead of the small handful it knew before. So "set up a streaming deck" produces a genuinely useful profile.

### ✨ Xenon AI — control your stream and per-app audio by voice/chat
- **Ask Xenon AI to run your stream**: when OBS, Twitch, YouTube or Streamer.bot are connected, the assistant can now act on them directly — *"start recording"*, *"go live"*, *"switch to my Gioco scene"*, *"set the title to…"*, *"change the game to…"*, *"clip that"*, *"shout out @channel"*, start/stop a YouTube broadcast, or trigger a Streamer.bot action. These tools appear only when the matching integration is set up, and if one isn't connected the assistant now tells you and points you to Settings → Streaming instead of silently doing nothing.
- **Per-app volume by voice**: *"lower Spotify"*, *"mute Chrome"*, *"turn Discord up"* — the assistant can adjust or mute a single application's audio without touching your master volume.

### 💅 Twitch widget — the chat is finally readable
- **The chat now keeps a clean, readable block by default** instead of being squeezed to a line or two under the tall stack of action controls. The actions (title, category, chat message, shoutout, chat mode, ad break) keep their room on a tall tile but gracefully shrink into their own small scroll area when space is tight, so the chat below always shows several messages at a glance.
- **Expanding the chat no longer hides your controls**: tapping the expand arrow used to swap the whole actions panel out for the chat. Now the chat takes over most of the tile *while the essential one-tap buttons — Go live, Mute, Clip, Marker — stay pinned right above it*, so you can read chat and still act on your stream without collapsing it again. The wordy title/category/chat-mode fields step aside and come back when you collapse.

### 🧠 Xenon AI — it actually knows everything it can do
- **The assistant is now explicitly aware of its full control surface** and reaches for the right control instead of just describing it. Its built-in briefing was previously a short, out-of-date list that left out whole areas it could already operate — RGB lighting, Performance Mode, page navigation, Deck profiles, live sensors, web search. It now has a complete, organised picture of everything it can do (audio/mic, media, notes/tasks/calendar/timers, RGB lighting, system stats and sensors, opening/closing apps, locking the PC, themes, pages, panels, screen vision — plus the opt-in Genesis/Streaming/Guardian tools when enabled), so requests like the streaming-page example above are carried out fully rather than partially. It's also told to say so plainly when something genuinely isn't one of its abilities, rather than pretending it did it.

## [v3.2.12] - 2026-06-30
### 🛠 Fixes — screen flickering (continued)
- **Less chance of the whole dashboard flickering while music is playing**, especially on hybrid-graphics laptops (those with both an Intel and an NVIDIA/AMD GPU). When a track is playing, the Media tile shows a soft blurred copy of the album cover behind it. That blurred layer was being kept permanently prepared on the GPU even when no music was playing — an always-on, full-size blurred surface that costs graphics memory for nothing and, on machines that switch between two GPUs, is exactly the kind of layer that can make the browser flicker the entire window. It is now prepared only while a cover is actually on screen and costs nothing when idle, which both lightens the load and removes a flicker trigger. The blurred ambient background glow was also isolated so that other things appearing on screen (like that cover) no longer force the browser to redraw the full-screen glow underneath them. This targets a Chromium-level flickering issue with blurred layers on dual-GPU setups; if it still flickers on such a machine, switching the dashboard's browser to a single GPU (Windows → Graphics settings) or turning off browser hardware acceleration is the remaining workaround.

## [v3.2.11] - 2026-06-30
### 🛠 Fixes — screen flickering
- **The dashboard no longer flickers when you have a custom image or video background**: the animated neon grid was being drawn *on top of* a custom wallpaper (only the softer "aurora" glow was correctly hidden behind one). On its own that's just visual clutter, but combined with the fix below it was also the most visible part of a flicker some people saw. The neon grid now hides itself whenever you set a custom background — exactly like the aurora already did — so your wallpaper (the forest video, a photo, anything you picked) shows cleanly with nothing animating over it. Turn off your custom background and the grid returns as before.
- **No more brief flicker while you close windows or Alt-Tab around the desktop**: the dashboard automatically pauses its animated background effects while a full-screen game is running ("game mode"). As you closed windows and tidied up, the focused window could momentarily *look* full-screen for a fraction of a second, which flipped game mode on and then straight back off — fading the ambient background out and in each time, seen as a flicker that stopped the instant you touched the dashboard (which brings it back into focus). Game mode now waits for the state to actually settle before reacting, so those split-second focus changes are ignored and the background stays steady. Pausing for a real game you start is unaffected. (Fixes the *"Screen flickering"* report.)

## [v3.2.10] - 2026-06-29
### 🛠 Fixes — Xenon AI voice input
- **The voice assistant no longer "goes quiet" while you're still mid-sentence**: a voice turn was force-stopped after a fixed **8 seconds**, so anything you said past that point was simply cut off — exactly the "sometimes it stops listening while I'm talking" report. The real end-of-turn is now driven by silence detection on the server, which stops the moment you actually finish speaking (about 2–3 seconds after you go quiet); the hard timer is now only a much longer **30-second** safety net for runaway recordings, so a normal multi-second sentence is never clipped. You can still stop it instantly at any time by tapping the screen, and tapping the orb restarts listening.
- **"Detected and active, but it doesn't hear me" no longer ends in a silent dead-end**: when a recording came back empty — a missed first word, a slow start, or a too-quiet mic — the whole voice session used to close without a word, which felt like a hang. Xenon now keeps listening and gives you another couple of tries with an on-screen *"Didn't catch that — try again"* hint, and only then ends the turn with a clear message pointing you to **Settings → Xenon AI** to check your microphone device and sensitivity. So instead of a mysterious silence you get a system that keeps listening when you mean to talk, stops when you're done, and tells you what to check if it genuinely can't hear you.
- **Switching your microphone now actually takes effect without restarting the server**: on setups that capture through DirectShow, Xenon bound to whichever microphone was the Windows default *when the server started* and never re-checked — so plugging in a headset and selecting it as your input did nothing, and recordings kept reading the old (often silent) device. Xenon now notices when your default microphone changes and re-binds its voice capture to the new device on the fly, so picking your headset just works.
- **New "Microphone test" button so you can confirm voice input works in 3 seconds** (Settings → Xenon AI): voice input records *server-side*, from a device that can differ from the one the Mic panel's level meter shows — which is why a mic could look "active" yet never be heard. The test runs that exact capture path for ~3 seconds and tells you plainly whether you were heard, **which device** was actually captured and at **what level (dB)**. If it didn't hear you it points you to fix it (pick another default mic, raise sensitivity). It's also the fastest way to report a problem: the result line is everything a maintainer needs in a single screenshot.

## [v3.2.9] - 2026-06-28
### 🛠 Fixes
- **The performance-optimization popup no longer keeps coming back after you dismiss it**: with automatic Performance Mode on, focusing a tracked app or game (OBS, a game, your editor…) opens the "optimize your PC" sheet so you can pick what to apply. Pressing **Cancel** hid it — but it didn't *remember* that you said no, so the moment that app regained focus the sheet popped up all over again, nagging you on every Alt-Tab back. Dismissing the sheet (Cancel or tapping outside it) now snoozes that activity for the rest of the session, exactly like the auto-suggest banner's "Ignore" already did, so it stays out of your way. Choosing **Apply** is unaffected, and the next fresh session still offers to optimize as before.

### ℹ️ Known issue — occasional soft rendering inside the iCUE widget at boot
On the Xeneon Edge the dashboard can occasionally look soft inside the iCUE widget — fuzzy text and thin lines — while the exact same dashboard stays razor sharp in a normal browser. We traced this all the way down and it is **not** a fault in the dashboard: it is an iCUE/QtWebEngine startup-timing issue. iCUE creates the widget's renderer at Windows logon, sometimes **before the Edge (a USB display) has finished being set up**, so the renderer is assigned the wrong display scale and the page is drawn at a size that doesn't match the panel, then resampled onto it — which is what smears the detail. That scale is then **frozen for the life of that renderer**, so a page refresh can't fix it (it reuses the same renderer and the same wrong scale).

The v3.2.9 attempt to correct this from inside the page (an auto-reload whenever the load came up mis-scaled) has been **removed**: we verified on-device that it cannot work — a refresh reuses the same renderer and therefore the same frozen scale, so it never recovers. (The earlier v3.2.8 page-zoom correction is left in place but only ever engages on a fractional-DPR wide-bar render and is a no-op at the panel's native scale; it cannot rebuild detail lost to resampling either.) In short: the render scale is set by iCUE and is not changeable or re-rolled from page-side code.

**Workaround that does work:** if the dashboard comes up soft after a PC restart, **remove and re-add the web widget in iCUE once** (re-creating it forces a brand-new renderer, which reads the now-stable display scale and renders crisp); it then stays sharp until the next restart. This is being reported to CORSAIR as the proper place to fix it (re-evaluate the web-widget's device scale *after* the USB display is enumerated).

## [v3.2.8] - 2026-06-25
### 🛠 Fixes
- **The dashboard no longer goes blurry when it runs inside iCUE on the Xeneon Edge**: in *iCUE Widgets* mode the dashboard could look soft and grainy — fuzzy text, pixelated thin lines and chart strokes — while the very same dashboard opened in a normal browser ("Desktop" mode) stayed razor sharp. The cause is that iCUE renders the page at a *fractional* display scale, so 1-pixel lines land between physical pixels and get smeared. Xenon already corrected for this by realigning the page to a whole-pixel scale, but it only did so **once, at startup** — and inside iCUE that scale isn't stable: it can read one way as the widget first loads and then shift after the widget settles, after the screen sleeps and wakes, or after a re-layout. So the correction got "stuck" on an outdated value, which is exactly why the blur came and went seemingly at random ("sometimes sharp, sometimes not"). The correction is now **re-evaluated live** whenever the display scale or the widget size changes, applying — or cleanly removing — itself to match the current state, so the dashboard stays crisp from the moment it loads and through sleep/wake and layout changes. It also now only nudges the scale when doing so genuinely sharpens the picture: on the common case where the display is already rendering near the panel's native resolution, it leaves the page completely alone instead of shrinking it (shrinking was softening the image rather than helping). Normal desktop and laptop browsers are untouched as before. (Fixes the *"Dashboard blurry in iCUE Widgets mode"* report.)
- **Thin curved outlines (the weather pill, the round chat buttons, the topbar buttons) look cleaner on the Xeneon Edge**: with the blur above resolved, a subtler artefact remained — those small *curved* 1-pixel borders could still look slightly jagged or broken up. This is because iCUE does a final, uneven rescale of the widget onto the panel that no page-side fix can fully undo, and a hairline thin line is the most fragile thing to put through it. Those few outlines now use a slightly more solid line (and the weather sun's soft glow is tightened) so they hold together through that rescale. This is a **device-only** refinement — it applies solely on the Xeneon Edge's wide fractional-scale display and changes nothing on a normal monitor.

## [v3.2.7] - 2026-06-24
### 🛠 Fixes
- **The one-click update finally works instead of always saying "Preparation failed"**: every in-app update since v3.2.4 stopped at *"Preparation failed — you can download manually"*, forcing a manual download. The cause was a typo in the release version itself — `package.json` shipped its version as `v3.2.6` (with a stray leading "v") while the safety check that validates a freshly-downloaded build expects plain `3.2.6`, so the two never matched and the update was rejected every time. The version is now clean semver again, and the validation tolerates a stray "v" on either side so this can't come back. **Why it matters:** because the auto-update kept failing, people downloaded releases by hand and, when replacing their folder, sometimes wiped their `server\data` — losing notes, Deck keys and settings. With the one-click update working again, the updater swaps files *in place* and never touches your data, so notes and Decks carry over automatically. Existing installs stuck on v3.2.4–v3.2.6 can update to this release one last time from the GitHub releases page; from here on, in-app updates work on their own.
- **The update popup no longer shows a doubled version like "from vv3.2.4"**: the same stray "v" leaked into the "Update available" dialog. Versions are now always rendered cleanly (`v3.2.6 · from v3.2.4`).
- **A failed update preparation now tells you *why***: instead of a blind "Preparation failed", the dialog now appends the reason (for example `version_mismatch` or `download_failed`), so a problem can be diagnosed from a screenshot instead of guessed at.
- **Deleting a page no longer leaves stale tile state behind**: removing a page already hides its widgets (restorable from the "+" palette), but a hidden widget kept the size and position it had on the now-deleted page — leftover saved state for a layout that no longer exists. Its geometry is now reset to the default on page removal, so nothing stale accumulates and a re-added widget gets a clean slot. (The layout already drops orphaned tab-groups, duplicated copies and invalid page references on every load, so dead state never piles up.)

## [v3.2.5] - 2026-06-23
- **Deleting a page now really removes everything on it** — including duplicated/tab-grouped tiles: previously, deleting a page hid its normal widgets and removed its duplicated *standalone* tiles, but any **tab-group** on that page (for example a duplicated Playback + Chat tile) was left behind and silently moved onto another page, so it kept reappearing no matter how many times you deleted. The dashboard now detects this and slides the stacked tile down into the first free space when the page loads, leaving the original tile and every other tile exactly where they were. Healthy layouts are untouched.


### ✨ New Features
- **Choose your clock format — 12-hour or 24-hour**: a new **Clock** option under **Settings → Appearance** lets you pick how the time is shown on the dashboard and the lock screen — **Auto** (follows your selected language, the previous behaviour), **12-hour** (with AM/PM) or **24-hour**. Until now the format was tied to the language with no way to override it, so an English user was stuck with AM/PM and couldn't switch to 24-hour. Your choice is saved and applies instantly to both clocks. (This also fixes a quiet inconsistency where the dashboard clock and the lock-screen clock could disagree on 12h vs 24h.)

### 🛠 Fixes
- **Your notes now reappear on their own after an update instead of looking blank**: occasionally after downloading a new version the Notes tile came up empty, even though the notes were perfectly safe — visible again as soon as you reopened the dashboard. The cause was timing: if the page reloaded a split second before the server had finished restarting, the one-time fetch of your notes failed and was never retried, so the box stayed empty until a manual refresh. Notes now retry loading on their own (with a short back-off) until they come through, so the tile repopulates by itself. Your notes were never lost — they live on the server — but now you won't have to reload to see them.
- **Reproducing a saved page no longer loses the System / Volume / Microphone tabs when you delete the other page**: if you saved a page that included the System tile and added it onto a new page, the live System panel (which physically hosts the Volume and Microphone tabs) only ever existed on one of the two pages. Deleting *that* page parked the live panel away and left the surviving page showing a dead, stats-less copy — your Volume and Microphone tabs vanished from the page you wanted to keep. Deleting a page now keeps the live System panel on the page you're keeping, so its Volume and Microphone tabs persist as expected. (Independent tiles like a Deck or Browser are unaffected — each copy keeps its own keys/address as before.)
- **Dropdown menus no longer get cut off at the edge of the screen**: on the short Xeneon Edge display, opening a dropdown whose list didn't fit (for example the Deck key editor's *LED reaction* "One-shot / Follows state" picker) could leave the options spilling past the bottom of the screen, where you couldn't reach them. Open dropdowns now always size and position themselves to stay fully on screen — scrolling inside the menu when the list is taller than the space available — so every option is reachable.
- **The one-click update no longer needs a Windows admin (UAC) prompt when Xenon is installed in your own folder**: the in-app updater always asked for administrator rights, which popped Windows' UAC dialog on the *primary* monitor — unreachable on multi-monitor or touchscreen setups like the Xeneon Edge, where users couldn't find or tap it and the update simply never proceeded. The updater now checks whether your install folder is writable (the normal case, when Xenon is unzipped to Desktop/Downloads): if so it applies the update directly with **no UAC prompt at all**. It only falls back to asking for administrator rights when the install really is in a protected location (e.g. Program Files). Note: this improvement takes effect for updates starting *from this version onward* — to get here, install this release once manually from the GitHub releases page.

## [v3.2.4] - 2026-06-17

### 🛠 Fixes
- **Deck hotkeys that use the Win key now actually fire** (Win+D, Win+E, …): after building a Win-based combo in the new composer, tapping the key did nothing. The Win key is a Windows "extended" key and was being sent without the flag that marks it as such, so Windows never saw a real Win press — only Ctrl/Alt/Shift combos worked. Win (and the navigation keys) are now sent correctly, so the whole composer works end to end.
- **The in-app update no longer gets stuck on "Updating…" with nothing happening**: when Xenon was running with administrator rights, the updater backed up your install and then silently stopped before swapping in the new files — leaving the dashboard frozen on "Updating…" and the server shut down. The cause: the already-elevated update helper kept running as a child of the app, so when the app stopped itself mid-update Windows killed the helper along with it. The helper now always relaunches itself as an independent process first, so the swap completes and the app restarts on its own. (Your data is backed up and automatically rolled back on any failure, so nothing is ever lost.)

### 🛠 Fixes
- **Recording a Deck hotkey no longer fires the shortcut while you set it up**: assigning a combo like **Win+D** used to trigger it the moment you pressed the keys (minimising your desktop) instead of just capturing it — because a web page can't stop Windows from acting on its own system shortcuts. The hotkey field is now a **composer**: tap the **Ctrl / Shift / Alt / Win** chips to choose modifiers, then press just the single main key (a lone key triggers nothing). Win+D becomes "tap *Win*, press *D*" — captured cleanly, never fired. Holding modifiers while you press the main key still works for safe combos like Ctrl+Shift+M.

The dropdown now lists every installed model under an **Installed** group, so you can select one directly; the list refreshes whenever the local AI panel opens.
- **A duplicated System panel now keeps its Sistema / Volume / Microfono tabs**: copying the System tile — or reproducing a saved page onto a new one — used to leave a stats-only panel with the Volume and Microphone tabs missing. Duplicates now carry the full tab bar with working tab switching, and every tab stays live (stats, network, volume and mic levels all mirror the original in real time). Each copy switches its own tabs independently of the original.
- **A duplicated Media player now keeps its proper layout instead of looking broken**: when you added a second Media tile, the artwork and the title/controls collapsed into a misaligned stack because the playback layout was tied to the original tile's internal name (which copies don't carry). The layout is now style-based, so every Media copy lays out exactly like the original — artwork on the left, track info and controls on the right.
- **Adding the Microphone (or Audio, Notes, Tasks, Calendar, Timer) to another page no longer makes it vanish from where it already was**: these widgets live inside a host tile by default — the Microphone and Audio sit in the **System** panel's tabs, and Notes/Tasks/Calendar/Timer in the **Agenda** panel — so adding one from another page's **+** menu used to *move* that single content over, leaving an empty gap where it had been. Adding it now creates an independent **copy** instead and leaves the original exactly where it was, so you really can have the mic (or any of these) in two places at once. Their live state (mute, level, lists, countdowns) stays mirrored across every copy as before.
- **No more duplicate "Genesis" toasts on top of the AI's own reply**: when you ask Xenon to build a dashboard, the assistant's reply already tells you what it created — but a *"Page created"* / *"Deck set up"* toast also popped up over the open AI overlay, overlapping the message. Those Genesis toasts are now shown only when the AI overlay is closed (for example when Genesis is triggered from a Deck key without the chat open), so they stay useful without doubling up on what the assistant just said.

### ✨ Improvements
- **Performance Mode settings are clearer, and Automatic mode can now ask before closing apps**: the two overlapping switches — *"Suggest automatically"* and *"Apply by itself"* — have been replaced by a single, easy-to-read **Mode** selector with three choices: **Manual** (only optimizes when you press *Optimize now*), **Suggest** (shows a prompt when an activity starts), and **Automatic** (applies the safe, reversible tweaks by itself and restores them when the activity ends). A short line under the selector explains what the chosen mode does, and the per-activity list is hidden in Manual mode where it doesn't apply. On top of that, **Automatic mode now opens the optimization panel when *"Offer to close background apps"* is on** — so even hands-off, you still get to choose which background apps to close (and review the Xenon AI plan, when AI is enabled) instead of that step being skipped silently; the resulting session is still treated as automatic and restores itself when you stop the activity. With app-closing off, Automatic mode stays fully silent as before.

## [v3.2.1] - 2026-06-15

### 🛠 Fixes
- **Fresh installs no longer end with "127.0.0.1 refused to connect"**: on a brand-new setup the installer could silently fail to install the widget's dependencies, so the server crashed the instant it started and the dashboard showed `ERR_CONNECTION_REFUSED` even though every other setup step looked successful. The cause was a Windows quirk: the installer asked the system for "npm" and got handed npm's PowerShell wrapper (`npm.ps1`), which — when launched through the command prompt — Windows *opened in Notepad* instead of running, so `npm install` never actually ran, the `node_modules` folder stayed empty, and the server (which needs the `ws` package just to start) exited immediately. The installer now runs npm reliably by calling Node directly, **checks that all three required packages are really present** before and after installing (instead of assuming success from an exit code), and clearly tells you to run `npm install` manually if anything still didn't land. Anyone who already hit this can simply re-run `INSTALL.bat` with this version, or run `npm install` once in the Xenon folder.

## [v3.2.0] - 2026-06-15

### ✨ New Features
- **New "Browser" widget — a live, interactive web page inside the dashboard**: add it from a page's **+** menu (under *System*) and you get a real, controllable browser tile right on the Xeneon Edge — type an address and browse, tap links, scroll and type, with Back / Forward / Reload and an **expand-to-fullscreen** button. Unlike a plain embed it works with sites that normally refuse to be framed (YouTube, Google, most web apps), because it's driven by a dedicated headless Microsoft Edge under the hood rather than an `<iframe>`. It's built to be light: a tile only does any work **while it's actually on screen** — switch to another page and it stops streaming immediately, and the background Edge shuts itself down once no Browser tile is open, so an added-but-unused widget costs nothing. You can add several Browser tiles (each remembers its own address), and the whole thing is local-only — the page stream never leaves your PC. *Notes for this first version: it needs **Microsoft Edge** installed (it is, on every Windows 11 — the widget shows a clear message if not), and the tile shows **video only, no sound** (the page is rendered headless for efficiency) — a known limitation we may revisit.*
- **New "Second screen" widget — a real extra Windows desktop inside the dashboard**: add it from a page's **+** menu (under *System*) to get a genuine second monitor you can see and control from the Xeneon Edge — drag any app onto it, then tap, scroll and type to drive it, with an **expand-to-fullscreen** button. It's **one-click to set up**: the tile installs a signed virtual-display driver and creates the extra screen for you (just accept the Windows prompts); if a step can't be automated — winget missing, or a restart needed to finish — it tells you exactly what to do rather than failing silently. Like the Browser tile it's built to be light: it only streams **while it's actually on screen** and **pauses automatically while you game** (the same *"Pause heavy tiles"* Performance option), so an open-but-unused tile costs nothing, and a static desktop barely sends any frames. Everything stays local — the screen never leaves your PC. You can fine-tune it in **Settings → Second screen** (smoothness up to **60 FPS**, image quality and the virtual display's resolution) — picking a resolution now applies it to the virtual monitor instantly, without re-installing anything or rebooting (including ultra-wide modes like 2560×720 so the view fills the Xeneon Edge bar with no side bars). When the virtual screen's shape doesn't match the tile, a **fit toggle** on the tile lets you choose between *filling the tile edge-to-edge* (cropping the overflow) and *showing the whole screen* (letterboxed) — your choice is remembered. The **mouse pointer is now drawn into the view**, so you can always see where you're pointing (the capture previously never included the cursor, which made the virtual screen feel uncontrollable). Touch behaves naturally on a large tile: by default a **swipe scrolls the dashboard** (so the tile no longer swallows your page swipes) while a **tap clicks** in the virtual screen — both at once, no mode switch. A toolbar **toggle** flips to *full control* when you need to drag inside the screen (window moves, sliders, text selection). The toolbar buttons were also enlarged for touch. The screen now also **persists properly**: it stays set up across restarts and automatically returns to your chosen resolution (Windows otherwise resets a virtual monitor to a tiny default after a reboot), so you never have to set it up again. When you're done with it, **Rimuovi schermo** removes the virtual monitor cleanly, now with a confirmation first. *Notes for this first version: it needs the **Xenon Helper** companion (installed by INSTALL.bat; the tile shows a clear message if it's missing), the live view is **video only, no sound**, and some hardware-accelerated/DRM-protected windows (certain video players) may show black — ordinary apps capture fine.*
- **Performance Mode now frees the Browser tile while you game**: a new option in **Settings → Performance** — *"Pause heavy tiles while gaming"* (on by default) — automatically suspends the Browser widget's live stream while a game is running or an optimization session is active, then resumes it on its own when you're done. Because the stream is what costs resources, pausing it (and letting the background Edge shut down) means an open Browser tile takes essentially nothing away from your game. Prefer to keep it live mid-game? Just turn the option off. This also covers the future second-screen tile.
- **Deck → Streamer.bot actions**: a new Deck action lets a key fire any of your **Streamer.bot** actions directly, so you can drive Twitch/YouTube/Kick automations, OBS scenes, sound effects, chat commands and everything else you've built in Streamer.bot from a Xenon Deck button — including a **second PC**, since Xenon connects out to wherever Streamer.bot is running (point the host at that PC's LAN address). ### 🛠 Fixes
- **A Deck key edit no longer gets lost after a restart — Deck saving rebuilt on a single source of truth**: the Deck used to be saved by every open dashboard (the Xeneon Edge display, a browser tab on the PC, …) pushing its *entire* copy of all Decks to the server, with "whose copy wins" decided by counters each one kept on its own. Those counters aren't comparable across devices, so a dashboard you hadn't used in a while could come back after a reboot, look "newest" by its own count, and overwrite everything — reverting a key you'd just changed and even wiping a Deck it had never seen. That whole mechanism is gone. Now the **server owns the saved Decks** and every dashboard sends **only the precise change you actually made** ("this key on this Deck changed", "this Deck was removed"), kept in a small local outbox and **retried until the server confirms it** — so an edit made while the server is restarting still lands, and a stale dashboard simply has nothing to send and can never overwrite or delete anything. Crucially, this now holds even for a dashboard still running the *previous* version's Deck code (e.g. a browser tab you left open before updating): its old "push everything" save is now accepted **additively** — it can only restore a Deck the server is missing entirely, never overwrite one that's already there — so it can no longer race in after a reboot and revert the button you just changed. Nothing changes in how you use the Deck. *(This release also fixes the deeper reason the loss kept happening: a Deck key with an **image icon** makes the saved data large, and the save was being sent in a "survives page close" mode that browsers cap at 64KB — so once a Deck held a couple of image icons, the save silently failed every time and the change only ever lived on the screen you edited it on. - **RGB on the cooler and fans now lights up after a reboot, not just the RAM**: with an **iCUE LINK** setup, the cooler and fans connect through the iCUE LINK System Hub — and right after a Windows restart iCUE reports its SDK session "ready" a moment before it has finished registering the devices on that hub. Xenon enumerated in that gap, found the RAM (a directly-connected module) but saw the hub as having **zero LEDs**, and never looked again — so music, album colours and effects lit only the RAM while the cooler and fans stayed dark until the next server restart. Xenon now notices an incomplete device list (a device reported with no LEDs, not just an empty list) and **re-checks for a few seconds** until iCUE finishes bringing the LINK devices online, then paints them — bounded so a hub that genuinely has no RGB attached doesn't keep retrying. For a perfectly fixed layout you can still turn **Auto** off in the Deck editor and set the columns/rows by hand. (Together with each Deck now being independent — see above — this closes the whole family of "the Deck rearranges itself" issues.)

### ⚡ Performance
- **New native helper (Xenon Helper)**: Xenon now ships an optional native companion, `xenon-helper.exe` (~11MB, no runtime to install), that replaces background PowerShell helpers with small native processes (~20MB instead of ~100MB each). Three moved in this release: the **media reader** — now-playing info served in ~2ms, and the helper *listens to Windows' media events*, so track changes and play/pause appear on the dashboard **instantly** instead of up to 2 seconds later — the **game-mode probe**, which now also reacts the instant you switch into or out of a game — and the **app-switcher probe** (see below). The helper is completely optional: if the exe is missing or anything goes wrong, Xenon silently uses the existing PowerShell paths, exactly as before. (Hardware sensor readings deliberately stay on the PowerShell path: the LibreHardwareMonitor library installed on users' machines can only run there.) `INSTALL.bat` downloads the helper automatically from the GitHub release — the exe is built by a public CI workflow from the tagged source, so its provenance is verifiable — and refreshes it on update if it's outdated; if the download fails, everything keeps working without it. `UNINSTALL.bat` removes it.
- **The Apps panel now opens near-instantly**: listing your open windows (with live previews and icons) used to launch a PowerShell process that *compiled C# code on every single open* — roughly a second of waiting before anything appeared, and the same again on every focus tap. The native Xenon Helper now does it directly (about 2× faster end-to-end, with no compilation step at all), and as always falls back transparently to the PowerShell path when the helper isn't installed. Re-run `INSTALL.bat` (or wait for the next release's auto-refresh) to get the updated helper.
- **No more PowerShell processes popping up every two seconds**: the now-playing info (track, artist, album art, progress) used to be read by launching a brand-new PowerShell process for every refresh — roughly one every 2 seconds while the dashboard is open. Each launch costs a full PowerShell startup, which showed up as `powershell.exe` flickering at the top of Task Manager and could measurably raise CPU usage and temperatures (reported by a user as ~5°C). Media is now read by a **single long-lived background helper** that stays connected to Windows' media system and answers each refresh in a few milliseconds, with the album art re-read only when the track actually changes. If the helper ever has a problem, Xenon falls back transparently to the old per-refresh method, so nothing breaks — it just uses a little more CPU until the helper recovers.
- **Sensor readings are ~10× cheaper**: the background helper that reads CPU temperature and GPU load/temperature used to rebuild its entire LibreHardwareMonitor session on every reading — re-scanning the disk for the library, re-enumerating all hardware and re-opening the kernel driver every few seconds, then querying three WMI fallbacks even when it already had a good value. The session is now opened once and kept alive (a reading went from ~500ms to ~30ms of CPU), fallbacks only run when the library really has nothing, and the nvidia-smi path and GPU name lookups are cached. If the session ever breaks (sleep/resume, driver reset), it rebuilds itself on the next reading.
- **Network panel readings rebuilt on lightweight APIs**: the ping/jitter/bandwidth collector runs every 3 seconds while the System panel is visible, and it was doing all three the expensive way — pings through WMI (the slow PowerShell 5.1 path), bandwidth through two more WMI queries, and — when no game was running — an FPS fallback that *slept 600ms inside the shared sensor worker on every poll*, delaying the CPU/GPU readings queued behind it. Pings and bandwidth now use direct .NET calls (same numbers, a fraction of the cost), the FPS fallback is skipped entirely when PresentMon is installed (its reading always took priority anyway), and concurrent polls from multiple open dashboards now share one reading — which also fixes a subtle bug where two dashboards polling together could corrupt each other's bandwidth delta.
- **Disk details no longer poll PowerShell every minute**: drive labels and filesystem types only change when you plug/unplug a drive, so they are now refreshed every 10 minutes instead of every 60 seconds (free-space numbers were always read live and still are). This was the last recurring PowerShell process on an idle server.
- **Remote Control status checks are now ~30× cheaper**: when Remote Control is set up, the dashboard checks its status every 15 seconds — and every check used to launch `winget` twice (a notoriously heavy command), a PowerShell process, and the Tailscale CLI, multiplied by every open dashboard (PC browser + Xeneon Edge). Now: whether Sunshine/Tailscale are *installed* is cached (it can't change behind Xenon's back — installing through Xenon refreshes it), the Windows service state is read with the native `sc.exe` (~10ms instead of a ~300ms PowerShell startup), and the whole status response is cached for a few seconds server-side so any number of open dashboards share a single probe. Status-changing actions (pairing, block/unblock, install) refresh the cache immediately, so the UI never shows stale results.

#### Performance & games
- **Performance Mode got a brain**: the optimization flow now works with **real measurements instead of guesses**. The confirmation sheet shows each background app's actual RAM next to its name; the AI planner receives the measured RAM/CPU of every open app plus the system memory pressure, so it picks the apps that genuinely cost something (and can say *why* in its explanation). Even **without AI**, a conservative built-in preselect now ticks the obvious wins for the current activity (e.g. office apps while gaming, game launchers while coding) — only when they actually hold real memory. After applying, a toast reports the **measured impact** ("~1.4 GB of RAM freed"). And the sheet **learns from you**: apps you repeatedly keep are never pre-ticked again, apps you always close get pre-ticked.
- **Fully automatic mode (opt-in)**: a new *"Apply by itself"* switch in Settings → Performance makes the safe, reversible tweaks (pause animations, high-performance power plan, priority boost) apply **on their own** the moment an enabled activity starts — and **restore on their own** ~45 seconds after it ends (the delay survives Alt-Tabs and loading screens). Closing background apps is never automatic: that always goes through the confirmation sheet. Default stays "suggest first".
- **Three new tracked activities**: besides gaming, coding and writing, Performance Mode now recognises **Streaming** (OBS/Streamlabs in the foreground — or OBS actually on-air, even while you're in another window), **Content creation** (Photoshop, Premiere, Blender, DaVinci Resolve, Unity, Unreal…) and **Meetings** (Teams, Zoom, Webex, Skype). Each has its own toggle and customizable trigger-app list, all off by default.
- **Windowed games are now detected**: game mode used to require a full-screen window, so borderless/windowed titles never triggered it. The detector now cross-checks the focused window against PresentMon's live frame data — when the window in focus is the process presenting real flip-model frames at game rates, it counts as a game. Focus + frames together keep the old false positives (browsers, video players, the dashboard itself) excluded.
- **Real in-game FPS (PresentMon)**: the FPS counter can now show the actual frame rate of the running game — including **exclusive-fullscreen** titles, which the previous DWM/LibreHardwareMonitor methods couldn't read (they only worked windowed/borderless, hence the frequent `N/D`). PresentMon is now **downloaded automatically by the one-click installer** (`INSTALL.bat`) into `server/presentmon/` and removed by `UNINSTALL.bat` — no manual setup. The server streams its ETW present data in the background and reports the busiest game's FPS (the elevated startup task provides the rights PresentMon needs). It degrades gracefully to the old DWM reading if PresentMon couldn't be downloaded.
- **Performance Mode (opt-in, transparent, reversible)**: a new profile under **Settings → Performance** that optimizes your setup on demand. It **notices what you're doing** — playing a game, coding, or writing — and can show a small **banner** offering to optimize for that activity (you choose which activities trigger it; games only by default, so it never nags during normal work). You decide **which apps count** for each activity too: every activity has an editable list you can **add your own apps to or trim** (defaults included), so the suggestion fires exactly when you want. You can trigger it any time with **Optimize now** in Settings or the new **Optimize performance** button right on the System tile. Before anything happens, a **confirmation sheet** opens where **you tick exactly what to apply** — pause animations, power plan, a gentle **priority boost for the app you're actually using** (e.g. the running game), and which background apps to close — all clearly labelled by risk; nothing runs until you confirm. What it does today: **pauses the dashboard's animations and animated backgrounds** (zero-risk, instant), switches Windows to a **high-performance power plan**, **nudges up the priority** of the detected active app (a safe AboveNormal, never the system-starving levels), and — when you opt in — offers to **close background apps you pick** to free up resources. It's **fully reversible**: it remembers your previous power plan, the boosted process, the closed apps' executables, and restores everything (power plan, priority, animations, and reopening the apps) when you hit **Restore** or end the session, even after a restart. **App closing is graceful** (the app can still prompt to save — nothing is force-killed), **never automatic**, and **critical Windows processes are always refused**. It's **off by default** and runs nothing without your explicit confirmation. A dedicated toggle lets you **use Xenon AI to drive the decisions** when it's configured, or keep them **deterministic** by turning AI off just for this mode — so Performance Mode works fully with or without AI. When AI is on, it **reads what you're doing** (gaming / coding / writing) and the apps you have open, then **pre-selects exactly which background apps are worth closing** and writes a one-line explanation at the top of the sheet — you still confirm before anything happens, and it only ever picks from apps that are actually open. With AI off, nothing is pre-selected and you choose entirely by hand. You can also just **ask Xenon AI** by voice or chat — "optimize performance" / "restore performance" — and it opens the same confirmation sheet (or undoes it). Works with both the Gemini and local (Ollama) providers.
- **Game mode (auto-pause ambient effects)**: the dashboard now detects when a real game is running full-screen and automatically **pauses the animated background** — the Aurora, the neon grid and the spinning glow of the Xenon AI button all freeze and fade out — so the widget stops competing with the game for the GPU. Everything resumes the moment you exit the game. Detection now keys on the **foreground full-screen window** (a game owns the foreground and covers its whole monitor with no title bar), which is far more reliable than the previous frame-rate-based approach: maximized desktop apps (they keep their title bar and leave the taskbar visible), the dashboard's own browser/WebView host, iCUE/Corsair and other apps that render continuously are all correctly excluded — so game mode no longer switches on by itself when no game is running. It needs no extra tools or administrator rights. A new toggle in **Settings → Performance → Game mode** turns this on or off (on by default). PresentMon remains an **optional** one-tap install, used only to show the real in-game FPS in the system panel — game mode itself works without it.

### 🔧 Reliability & Performance
- **Lower idle CPU usage**: the live sensor readings (GPU, CPU temperature, network) used to launch a brand-new PowerShell process on every refresh — the single biggest ongoing cost while the dashboard is open. They now share one long-lived helper process, so that startup cost is paid once instead of several times a second. Warm readings drop from ~150ms+ of process startup each to tens of milliseconds. The now-playing reader is intentionally left as-is (it must release media-session handles by exiting), and if the helper ever stalls the app silently falls back to the old per-reading method, so nothing breaks.
- **Hardened against cross-site drive-by control**: the quick-action endpoints that change state (mic mute, master/app volume, speaker mute) now reject cross-site requests, so a malicious web page you happen to visit can no longer silently toggle your microphone or volume through your local server. The dashboard itself is unaffected — it only ever talks to the server from its own page.
- **Faster dashboard reloads**: the dashboard's scripts and stylesheets are now served with proper revalidation (ETag) instead of being re-downloaded in full on every load. The browser still picks up any change immediately, but unchanged files are confirmed with a tiny `304 Not Modified` instead of resending the whole file — noticeably lighter on the Xeneon Edge's built-in browser, especially on repeated reloads.
- **Hardened request handling**: incoming request bodies are now bounded by a generous size limit, so a misbehaving or looping local client can no longer grow the server's memory without limit. Normal usage (including AI chat with screenshots) is unaffected.
- **Smoother "live" indicators**: the pulsing dots on the OBS / Twitch / YouTube live badges and the feed-refresh spinner now hint the browser to render them on their own compositor layer, keeping these always-on animations crisp without nudging the rest of the tile.
- **Lighter background work**: the app-icon cache is now bounded (long sessions can't grow it without limit), and the disk monitor no longer probes every possible drive letter each cycle — it remembers which letters are real and re-checks the full range only once a minute, so a newly plugged-in drive still shows up promptly.
- **Korean / Japanese / Chinese**: the "Start voice chat" label is now translated (it previously fell back to English in those three languages).
